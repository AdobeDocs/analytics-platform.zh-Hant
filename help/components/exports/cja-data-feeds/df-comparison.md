# 比較Customer Journey Analytics和Adobe Analytics中的資料摘要

Customer Journey Analytics和Adobe Analytics中的資料摘要可讓您將原始資料匯出至協力廠商平台。 如果您先前在Adobe Analytics中使用過資料摘要，請使用下列資訊來瞭解可用功能、資料處理行為等的差異。

## 比較資料摘要功能

| **組態選項和概念** | **說明** | **Customer Journey Analytics** | **Adobe Analytics** |
|---------|----------|---------|---------|
| 資料輸入 | 可收集並包含在資料摘要中的資料型別。 | 支援跨管道資料輸入，包括網路資料、客服中心資料、銷售點資料等。 | 僅支援網頁和行動資料輸入。 |
| 資料處理 | 通常需要先處理資料，才對報表有用，而這項處理會影響資料摘要功能。 系統會根據您使用的是Customer Journey Analytics還是Adobe Analytics，以不同階段處理資料。 | 資料會在報表時處理，因此Customer Journey Analytics中的許多報表功能都可用來變更歷史資料，例如拚接、衍生欄位、「資料準備」、細分和報表視窗。 | 資料會在收集時處理，因此處理規則、VISTA規則、拼接和分段等報表功能不會影響歷史資料。 |
| 資料摘要視窗/傳送頻率 | 決定：<ul><li>資料摘要的傳送頻率</li><li>資料摘要中包含的時間範圍<p>換言之，在資料摘要視窗結束並傳送資料摘要後，就會開始新的資料摘要視窗。</p></li></ul><p>提供下列選項：</p><ul><li>**每日**：摘要包含一整天的資料，從報表套裝時區的午夜到午夜。 此選項可用於回填或歷史資料，或用於繼續摘要。</li><li>**小時**：摘要包含一個小時的資料量。 使用此選項繼續摘要。</li></ul> | 每小時和每日摘要。<p>資料摘要視窗包含在&#x200B;**[!UICONTROL 報告視窗]**&#x200B;中</p>  <p>範例： 1小時</p> | 每小時和每日摘要。 此外，也提供15分鐘摘要，但預設無法使用。 </p><p>可根據&#x200B;**[!UICONTROL 回顧期間]**&#x200B;設定延遲送達點選。</p><p>範例： 1小時</p> |
| 納入延遲送達點選 | 延遲送達點選是指在目前資料摘要視窗中抵達，但時間戳記落在先前資料摘要視窗中的點選（例如，透過時間戳記點選或資料來源）。 當點選延遲到達時，即使點選的時間戳記屬於較早的視窗，點選也會批次處理到要傳送的下一個資料摘要檔案中。 | 一律包含發生在&#x200B;**[!UICONTROL 報告期間]**&#x200B;內的延遲送達點選。 <p>點選會自動依輸出中的時間戳記重新排序（例如，如果時間戳記較早，則較晚的點選會出現在準時點選之前）。 沒有變更摘要，因此會保留原始值。 | 可包含或排除延遲送達點選，並可使用&#x200B;**[!UICONTROL 延遲送達點選]**&#x200B;組態選項設定。<p>這些點選的回顧期間是透過可用於此特定用途的&#x200B;**[!UICONTROL 回顧期間]**&#x200B;組態選項設定的。</p><p>點選會以接收順序顯示；點選不會根據時間戳記重新排序。</p> |
| 順序錯亂的點選 | H | 系統會評估每個人的順序錯亂；CJA可以接受串流+批次抵達模式，並在資料到達適當時間序列時進行重新排序。 CJA不會等到確認已具有報表回溯期的所有資料後才傳送，而是會傳送當天到達的專案（可能包括報表回溯期內，時間戳記較早的延遲點選）。 <p>**重要**：資料摘要資料的最終使用者必須能夠處理順序錯亂的時間戳記，因為資料摘要資料可能包含時間戳記早於資料摘要視窗的延遲點選。</p> | |
| 回顧期間（僅限Adobe Analytics） | 在選取的&#x200B;**[!UICONTROL 傳遞頻率]**&#x200B;以外傳遞（例如&#x200B;**[!UICONTROL 小時]**&#x200B;或&#x200B;**[!UICONTROL 天]**）的延遲或順序錯亂點選的截止日期。 造訪必須在此截止日期之後開始才能納入；在截止日期之前開始並在回顧期間內結束的造訪不包括在內。<p>不用於持續性、工作階段或維度（這些會包含在收集的原始資料中）。</p> | 包含在&#x200B;**[!UICONTROL 報告期間]**&#x200B;中。 | 支援<p>範例： 23小時</p> |
| 報告期間（僅限Customer Journey Analytics） | 時間範圍包括：<ul><li>目前的資料摘要視窗（**[!UICONTROL 傳遞頻率]**&#x200B;欄位中選取的最近小時或日期）。</li><li>目前資料摘要視窗之前的指定時間，可允許任何延遲或順序錯亂的點選。</li></ul> <p>工作階段、持續性和區段需要。</p><p>不用於維度。 根據維度的配置和有效期，每個維度皆可控制維度。 Dimension回顧不可超過報表回溯期。</p> | 支援<p>範例： 24小時</p> | 不適用<p>請參閱「回顧期間」</p> |
| 初始回填視窗 |  | 範例： 7天 | 範例： 7天 |
| 點擊 |  | 資料摘要視窗中只有點選5。 不過，由於報告時段也包含點選4和點選3 （它們是包含來自先前資料摘要時段之時間戳記的延遲送達點選），因此它們也包含在目前的資料摘要時段中。<p>點選會根據其時間戳記，在資料摘要中重新排序，如下所示：點選3、點選4、點選5。</p> | 資料摘要視窗中只有點選5。 不過，由於已設定回顧，且包含點選4和點選3 （即具有來自先前資料摘要視窗之時間戳記的延遲送達點選），因此這些點選也會納入目前的資料摘要視窗中。 （如果未設定回顧，資料摘要中只會包含點選5。） <p>點選會依其接收順序顯示在資料摘要中，如下所示：點選4、點選3、點選5。</p> |
| 延遲送達點選/順序錯亂點選 | (在AA中：資料順序錯亂表示：每位訪客的順序錯亂。 在CJA中：這表示每個人的訂單混亂。 如果您傳送每人順序不正確的資料給我們，前提是您設定時間戳記。 您可以兩種方式設定時間戳記：讓Adobe根據我們收到資料的時間設定時間戳記。 或者，您也可以自行設定。 如果您設定時間戳記，而您傳送的資料順序有誤，這會在AA中造成問題。 在AA中，資料必須依每個訪客的順序排列。 我們需要正確的事件順序。 但在CJA中，資料上的時間戳記並不重要。 CJA不會為點選指派時間戳記。 這是上游完成的。 CJA在資料到達後會重新排序資料，讓所有資料都處於正確的時間順序。 然後我們就可以進行報表時間處理了。 這表示您可以同時擁有串流資料和批次資料。 沒關係。 到達時，我們會重新排序，因此會變成依每人排序。 所以在CJA中，我們會提供您過去一天或一小時收到的所有資料，但僅限於報告期間的開始。 最有可能的情況是，一天或一小時內取得的大量資料，都屬於當天或一小時。 如果您只是從客服中心批次處理資料，那麼您就會得到這些資料。 在CJA中，資料可以進入，但何時進入並不重要。 所以資料摘要客戶必須能夠自行處理。 所以無論使用者將資料放在何處，都需要處理時間戳記可能遍佈所有位置的事實。 對於某些客戶而言，這可能是一項挑戰。 他們需要知道這一點。 必須能夠處理每個人的順序錯亂資料。 跨人皆不重要。 )點選應包含在先前的資料摘要中，但由於某些原因（例如透過時間戳記點選或資料來源）延遲到達。 <p>即使延遲送達點選的時間戳記位於先前的資料摘要期間內，這些延遲送達點選仍會在到達時包含在目前的資料摘要中。 每次資料摘要處理資料時，都會檢視任何已送達的延遲點選，並將它們批次放入下一個已傳送的資料摘要檔案中。</p> | 一律包含發生在&#x200B;**[!UICONTROL 報告期間]**&#x200B;內的延遲送達點選。 <p>這些延遲送達點選的回顧期間是透過&#x200B;**[!UICONTROL 報告期間]**&#x200B;組態選項所控制。</p><p>點選會根據時間戳記自動重新排序；原始值會持續存在（無變更摘要）。</p> | 可包含或排除。 可使用&#x200B;**[!UICONTROL 延遲送達點選]**&#x200B;組態選項設定。<p>這些點選的回顧期間是透過可用於此特定用途的&#x200B;**[!UICONTROL 回顧期間]**&#x200B;組態選項設定的。</p><p>點選會以接收順序顯示；點選不會根據時間戳記重新排序。</p> |
| 區段 |   | 您在Customer Journey Analytics中設定的區段可套用至資料摘要或資料檢視。<p>（人員、工作階段、事件）「人員」容器會使用整個報告期間；可以根據區段邏輯展開DF點選輸出。</p> | 無法套用區段。 |
